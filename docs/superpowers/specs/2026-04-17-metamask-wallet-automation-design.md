# MetaMask 钱包自动化设计方案

**日期：** 2026-04-17

**状态：** 已确认，可进入实施规划

## 目标

为 TestHub 增加第一期钱包自动化能力，使用用户本机真实的 Chrome Profile 执行 EVM dApp 流程，并在同一条测试链路中支持以下 4 类钱包动作：

- 连接钱包
- 切链
- 签名消息
- 发送交易并在钱包中确认

## 范围

本方案覆盖：

- 仅支持 Windows 本地部署
- 仅支持 Google Chrome
- 仅支持用户自己的 Chrome Profile
- 仅支持 MetaMask
- 仅支持 EVM dApp
- 仅支持有头执行
- 集成到现有 UI 自动化 AI 执行链路中

本方案不覆盖：

- Firefox、Edge、WebKit 的钱包自动化
- Phantom、Backpack、OKX Wallet 或多钱包路由
- 钱包创建、导入、密码解锁、助记词管理
- CI 执行或远程浏览器环境
- Solana 或非 EVM 链

## 产品结果

功能上线后，用户可以在 AI UI 自动化任务里开启钱包模式。TestHub 会先关闭现有 Chrome 窗口，再使用用户配置的 Chrome Profile 以受控远程调试会话方式重新拉起 Chrome，执行 dApp 页面流程，并以确定性方式处理 MetaMask 弹窗，完成首期支持的 4 类钱包动作。

## 设计原因

当前 AI 浏览器链路不足以支撑钱包自动化，原因很明确：

- 当前 AI 浏览器模式会禁用扩展
- 当前执行链路把所有交互都当作通用页面动作处理
- 钱包弹窗属于高风险、强状态、强版本相关场景

更稳的边界应该是：

- 现有 AI/UI 自动化继续负责 dApp 页面
- 新增专门的钱包适配层负责 MetaMask 弹窗
- 新增受控 Chrome Profile 会话启动层负责浏览器接管

这样既保留了“使用你自己浏览器环境”的体验，也避免把 MetaMask 弹窗完全交给 AI 自由发挥而导致不稳定。

## 架构

实现时会在现有 UI 自动化链路外增加 3 层能力。

### 1. 受控 Chrome 会话层

这层负责以可预测方式接管用户真实的 Chrome Profile。

职责：

- 校验 Chrome 可执行文件路径
- 校验 `user_data_dir` 和 `profile_directory`
- 按配置决定是否关闭现有 Chrome 进程
- 以远程调试模式重新启动 Chrome
- 将自动化运行时接入到这个浏览器会话
- 校验目标 Chrome Profile 中存在 MetaMask 扩展

这层必须是显式且快速失败的。如果 Chrome 无法被受控接管，则 AI 任务在开始前就直接终止。

### 2. 钱包适配层

这层负责把钱包动作从“通用点击”升级成“确定性动作执行”。

职责：

- 检测 MetaMask 弹窗或新标签页
- 识别当前 MetaMask 页面属于哪种动作场景
- 执行支持的钱包动作
- 返回结构化成功或失败结果
- 写入详细执行日志

第一期的钱包适配器为 `MetaMaskAdapter`。它的接口需要保留可扩展性，方便未来接入 `OkxWalletAdapter`、`PhantomAdapter` 等，但这些不在本期范围内。

### 3. 现有 AI/UI 自动化层

这层继续负责：

- dApp 页面导航
- 表单输入
- 点击 dApp 控件
- 断言
- 截图与日志
- 自然语言任务拆解与执行

核心变化在于调度方式：当执行链路识别到钱包步骤时，流程临时切换到钱包适配层，动作完成后再回到 dApp 页面继续执行。

## 执行流程

钱包模式下的整体执行流如下：

1. 用户在 AI 测试页面启用钱包模式。
2. 前端提交 AI 任务时携带钱包模式元数据。
3. 后端读取当前启用的钱包浏览器配置。
4. 后端执行受控 Chrome 会话准备。
5. 后端验证当前 Chrome Profile 中已安装 MetaMask。
6. 后端启动或复用已接入的浏览器会话。
7. 现有 AI/UI 自动化执行 dApp 页面步骤。
8. 当运行到钱包步骤时，切换到钱包适配层。
9. 钱包动作成功后，把焦点切回 dApp 页面。
10. 流程继续执行，直到通过、失败或停止。

## 支持的钱包动作

第一期只支持以下动作。

### 连接钱包

预期行为：

- 识别 dApp 的连接钱包流程
- 在钱包选择器中选择 MetaMask
- 捕获 MetaMask 弹窗
- 使用当前默认选中账号完成授权连接
- 返回 dApp 页面并校验连接状态

幂等规则：

- 如果 dApp 已经连接 MetaMask，则直接记为成功并继续执行

### 切链

预期行为：

- 支持 dApp 发起的切链请求
- 支持 `wallet_addEthereumChain` 触发的“先加链再切链”流程
- 在 MetaMask 中确认切链
- 返回 dApp 页面并校验目标链

幂等规则：

- 如果当前链已经等于目标链，则直接记为成功并继续执行

### 签名消息

预期行为：

- 检测 MetaMask 签名弹窗
- 确认页面上存在可签名请求
- 点击签名确认
- 返回 dApp 页面并校验预期签名结果

### 确认交易

预期行为：

- 检测 MetaMask 交易确认弹窗
- 等待确认按钮区域稳定
- 点击确认发送
- 返回 dApp 页面并校验交易进行中或完成信号

## 配置设计

本功能扩展现有配置中心，不新开顶级模块。

### 配置位置

前端：

- `/configuration/ai-mode`

后端：

- `apps/ui_automation/views_config.py` 现有 UI 自动化配置视图区域

### 必要配置项

第一期需要一份启用中的 MetaMask 浏览器配置，字段如下：

- `enabled`
- `chrome_executable_path`
- `user_data_dir`
- `profile_directory`
- `remote_debugging_port`
- `force_close_existing_chrome`
- `metamask_extension_id`

行为规则：

- `profile_directory` 默认值为 `Default`
- `force_close_existing_chrome` 默认值为 `true`
- `metamask_extension_id` 支持自动探测，也支持手工覆盖
- 钱包模式下不允许无头执行

## 前端交互设计

本方案刻意保持前端改动轻量，不增加一整套新业务页面。

### 配置中心

现有 AI 模式配置页增加一块 `MetaMask 钱包浏览器配置` 区域，支持：

- 查看当前钱包浏览器配置
- 编辑 Chrome 可执行文件路径
- 编辑 Chrome 用户数据目录
- 编辑 Chrome Profile 目录名
- 编辑远程调试端口
- 启用或关闭“执行前关闭现有 Chrome”
- 执行配置可用性测试

配置可用性测试必须校验：

- Chrome 可执行文件存在
- Chrome Profile 路径存在
- 远程调试端口可用或可被占用接管
- 受控启动后可识别到 MetaMask 扩展

### AI 测试页面

现有 AI 测试页增加一个钱包模式控制区，包含：

- `启用 MetaMask 钱包模式`
- `执行前关闭现有 Chrome`
- 可选目标链输入或选择器
- 明显告警：本次运行会关闭现有 Chrome，并使用你的真实 Chrome Profile 重启浏览器

页面不直接暴露底层钱包动作按钮。用户仍然通过自然语言描述任务，后端在执行时识别钱包节点并切换到确定性钱包动作。

## 后端设计

### 新增领域对象

实现阶段建议补充以下模型边界。

推荐模型名与职责：

- `WalletBrowserConfig`：保存 Chrome 可执行路径、用户数据目录、Profile 目录、远程调试端口、扩展 ID、启用状态等配置
- `WalletSession`：保存一次受控 Chrome 启动及浏览器接入过程的运行态元数据
- `WalletActionLog`：保存结构化钱包动作结果，并挂接到已有 `AIExecutionRecord`

边界必须保持清晰：

- 配置存储
- 运行时会话跟踪
- 钱包动作审计日志

### API 分组

后端应提供 3 组 API。

#### 1. 钱包浏览器配置 API

用途：

- 钱包浏览器配置的增删改查
- 配置可用性测试
- 当前激活配置查询

#### 2. 钱包会话准备 API

用途：

- 按配置关闭现有 Chrome
- 以远程调试模式重启 Chrome
- 接入浏览器会话
- 验证 MetaMask 可用性

这组 API 属于调度型接口，不应作为通用浏览器工具随意暴露。

#### 3. 钱包动作运行时 API

用途：

- 内部执行 `connect_wallet`
- 内部执行 `switch_chain`
- 内部执行 `sign_message`
- 内部执行 `confirm_transaction`

这些动作通常不直接由前端按钮逐个调用，而是由 AI 执行记录内部统一触发。

## 弹窗与窗口处理

钱包自动化能否稳定，关键在于弹窗处理。

受控会话层必须具备这些能力：

- 检测新打开的 Chrome 标签页或弹窗窗口
- 区分 dApp 页面上下文与 MetaMask 上下文
- 正确切换控制目标
- 钱包动作完成后切回 dApp 页面

实现上不能只依赖 AI 看文字猜弹窗，而应结合显式浏览器上下文发现和确定性页面匹配规则。

## 错误处理

本功能必须做到动作级别明确失败，不允许模糊失败。

至少需要覆盖以下失败类别：

- Chrome 关闭失败
- Chrome 启动失败
- 远程调试连接失败
- Chrome Profile 路径无效
- MetaMask 扩展不存在
- MetaMask 弹窗未出现
- MetaMask 页面类型无法识别
- 钱包当前为锁定状态
- 账号授权被拒绝
- 切链失败
- 签名被拒绝
- 交易被拒绝

失败策略：

- 开启钱包模式后，不允许静默回退到通用 AI 点击模式
- MetaMask 预检查失败时，不允许降级为普通非钱包执行
- 所有失败细节都必须写入 `AIExecutionRecord`

## 边界与前置假设

本期明确依赖以下假设：

- 运行环境是 Windows
- 浏览器是 Chrome
- 用户接受执行前关闭所有 Chrome
- MetaMask 已安装在目标 Chrome Profile 中
- MetaMask 在运行前已由用户手工解锁
- dApp 采用标准 MetaMask 连接流程

如果上述任一条件不满足，执行应直接以前置检查失败结束。

## 安全与用户保护

该方案直接使用用户真实浏览器 Profile，因此必须把风险边界说清楚。

需要具备以下保护措施：

- 钱包模式必须显式开启
- 首次运行前页面必须展示风险提示
- 配置检测流程不得发送真实交易
- 真实交易确认必须在用户明确发起钱包模式任务的前提下执行
- 日志中不得保存助记词、私钥或钱包密码

本期不尝试自动化钱包创建、导入或解锁，避免进入敏感秘密管理范围。

## 测试策略

实施计划必须覆盖 3 类测试。

### 单元测试

目标：

- Chrome Profile 配置校验
- 钱包动作路由
- MetaMask 弹窗检测辅助逻辑
- 切链与动作结果解析

### 集成测试

目标：

- 钱包模式请求创建
- 受控 Chrome 会话启动
- AI 执行链路中的钱包动作委派

这类测试尽量使用桩或可控浏览器夹具，不要求每条测试都依赖真实钱包。

### 人工验证

第一期必须完成的人工验证：

- 在 EVM dApp 上连接钱包
- 从一条 EVM 链切换到另一条 EVM 链
- 完成一次登录或鉴权签名
- 发起一笔交易并在 MetaMask 中确认
- 验证缺少 MetaMask 时的失败行为
- 验证 MetaMask 锁定时的失败行为

## 发布顺序

该功能应通过显式钱包模式开关进行隔离，避免影响现有 AI UI 自动化用户。

建议发布顺序：

1. 受控 Chrome 会话启动能力
2. 配置中心与配置可用性检测
3. MetaMask 连接钱包流程
4. MetaMask 切链流程
5. MetaMask 签名流程
6. MetaMask 交易确认流程
7. 基于真实 EVM dApp 的端到端验证

## 后续扩展

以下内容明确延后：

- OKX Wallet、Phantom、Backpack 等其他钱包
- 多钱包选择与竞争弹窗处理
- 多 Profile 自动切换
- Edge Profile 支持
- CI 友好的独立自动化浏览器环境
- Solana 支持
- 钱包解锁自动化

## 验收总结

本期完成的判定标准如下：

- TestHub 能使用用户配置的真实 Chrome Profile 启动受控浏览器会话
- TestHub 能在执行前确认 MetaMask 存在
- AI UI 自动化任务在钱包模式下，能对 4 类支持的钱包动作走确定性处理，而不是交给通用 AI 弹窗点击
- MetaMask 在标准 EVM dApp 上的连接、切链、签名、交易确认流程可执行
- 失败场景具备明确分类、详细日志和动作级错误信息
